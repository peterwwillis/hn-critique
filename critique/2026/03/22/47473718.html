The three pillars of JavaScript bloat (43081j.com)

289 points by onlyspaceghost 8 hours ago  | 144 comments critique  | original on HN

Summary

The article discusses three main causes of JavaScript dependency bloat: legacy runtime support for very old engines, an atomic package architecture that fragments code into many tiny packages, and outdated ponyfills that remain in use after native support is widespread. It argues that these practices lead to unnecessary complexity, duplication, and security risks in modern JavaScript projects, and suggests community efforts and tooling to reduce bloat.

Main Points & Arguments

  • Legacy support for very old JavaScript engines (like ES3) leads to many small utility packages that are mostly unnecessary for modern environments.
  • Atomic architecture breaks code into extremely small packages, which often have only one consumer, causing duplication and increasing supply chain risks.
  • Ponyfills, which are non-mutating polyfills used by libraries, often remain in dependencies long after native support is available, adding unnecessary bloat.
  • Most developers today use modern environments that do not require these legacy compatibility layers, yet the bloat persists in common dependency trees.
  • Tools like e18e CLI, knip, npmgraph, and the module replacements project can help identify and reduce unnecessary dependencies.
