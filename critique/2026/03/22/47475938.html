Main Points & Arguments

  • Windows native app development has a long history of evolving UI frameworks, from Win32 to MFC, WinForms, WPF, WinRT, UWP, and now WinUI 3, each with compatibility and capability gaps.
  • Developers face difficult choices between C++, .NET framework-dependent deployment, and .NET AOT, with none offering an ideal experience due to issues like outdated preinstalled .NET versions and large binary sizes.
  • Key app features such as global keyboard shortcuts, tray icons, and certain window behaviors require falling back to older Win32 APIs via P/Invoke, complicating development.
  • Microsoft's packaging and deployment system (MSIX) is cumbersome and costly for independent developers, especially due to expensive code signing certificates and restrictive store policies.
  • The C# language and interop tooling (e.g., CsWin32) have limitations that hinder seamless integration with Windows APIs, and modern language features to reduce boilerplate in UI binding remain lacking.
  • Microsoft appears to deprioritize native Windows app development, with many first-party apps using web technologies and community efforts shifting to third-party cross-platform frameworks or Electron.
  • Despite some recent improvements in the Windows App SDK and WinUI 3, the ecosystem remains fragmented and frustrating, leading many developers to prefer web-based solutions.
