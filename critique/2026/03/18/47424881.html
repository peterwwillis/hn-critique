Summary

The article explains how Linux handles virtual memory, page faults, and the mmap system call, focusing on how userfaultfd enables userspace to handle page faults for faster virtual machine (VM) snapshot restores. It details the limitations of mmap overlays for lazy loading and how userfaultfd preserves mapping identity, allowing efficient on-demand page population during VM restore.

Main Points & Arguments

  • Linux uses virtual memory with demand paging, where physical pages are allocated only on first access, triggered by page faults.
  • Different mmap flags (anonymous private/shared, file-backed private/shared) affect how virtual addresses map to physical pages and their behavior.
  • Replacing an existing mapping with mmap and MAP_FIXED changes the physical pages behind the virtual address, breaking subsystems relying on physical page identity.
  • Virtual Machine Monitors (VMMs) use mmap to allocate guest memory and rely on KVM, VFIO, and vhost-user subsystems that depend on stable physical page mappings.
  • Userfaultfd allows userspace to intercept and handle page faults, populating pages on demand without changing the existing mapping, preserving physical page identity.
  • Using userfaultfd for VM snapshot restore enables near-instant VM start by lazily loading guest memory pages on access, significantly reducing restore latency compared to eager copying.
  • Concurrent faults, kernel feature requirements, and handler throughput are important considerations for implementing userfaultfd-based lazy restore.
  • On-demand paging changes I/O patterns and can reduce peak bandwidth usage but may introduce latency unpredictability under heavy concurrent faults.
  • Firecracker is a prominent VMM using userfaultfd with an external fault handler process to optimize snapshot restore.
  • Trade-offs include per-page fault overhead, random vs sequential reads, complexity for orchestrators, and kernel feature availability.
